# Field Tracking — Mobile (React Native + TypeScript)

The salesperson app. Signs in, requests location + background + notification
permissions, then runs an **adaptive background tracker** as a foreground service
that uploads GPS points to the backend and survives app-minimise / screen-lock.
Tracking stops only on sign-out.

> This folder contains the **JavaScript/TypeScript source and config**. The
> native `android/` and `ios/` projects are generated by React Native and are not
> checked in here — follow the bootstrap steps below to create them, then drop
> this `src/` + root files in.

## What's implemented

- **Auth** — JWT access/refresh, tokens persisted, transparent refresh on 401
  (`src/api/client.ts`), salesperson-only gate.
- **Permissions** — fine → background (correct Android 10+ ordering) →
  notifications (`src/services/permissions`).
- **Adaptive tracking** — `src/services/location/BackgroundTracker.ts`:
  30s cadence while moving, 2–5 min while stationary, decided by speed +
  displacement. Runs in a foreground service via `react-native-background-actions`.
- **Offline-first** — every fix is written to a durable queue
  (`src/services/offline/OfflineQueue.ts`) and only removed after the server
  acknowledges it. Each point carries a `clientId`, so resync after an outage
  cannot create duplicates. Survives app restarts.
- **Simple dashboard** — tracking status, inside/outside office, today / week /
  month distance + travel time, left/returned office times. No maps, per spec.
- **State** — Redux Toolkit (`src/store`).

## Bootstrap a native project

```sh
# 1) Create a fresh RN 0.76 TS app somewhere
npx @react-native-community/cli@latest init FieldTrackingMobile --version 0.76.5

# 2) Copy this folder's source over the generated app:
#    App.tsx, index.js, app.json, babel.config.js, metro.config.js,
#    tsconfig.json, and the entire src/ directory.

# 3) Install dependencies (from package.json here)
npm install

# 4) Native config
#    - Apply native-setup/ANDROID.md to android/
#    - Apply native-setup/IOS.md to ios/  (then: cd ios && pod install)

# 5) Point the app at your backend
#    Edit API_BASE in src/api/client.ts
#      device on LAN     -> http://<your-pc-ip>:4000
#      Android emulator  -> http://10.0.2.2:4000
#      iOS simulator     -> http://localhost:4000
```

## Run

```sh
npm run android      # or: npm run ios
```

Sign in with a salesperson account that the admin created from the dashboard's
**Team** page (the admin hands each salesperson their email + password).

## Verifying the typed source

The TypeScript here is written against the dependency types in `package.json`.
Run the type check after `npm install` in a real RN project:

```sh
npm run typecheck
```

(It cannot be type-checked without the native React Native packages installed,
since nearly every module imports from `react-native` and the native libraries.)

## Notes on background reliability

`react-native-background-actions` (MIT) gives a free, reliable Android foreground
service and works with iOS background location modes — sufficient for this use
case. For the most battle-tested behaviour across aggressive OEM battery savers
(Xiaomi/Oppo/etc.), teams often switch the engine in `BackgroundTracker.ts` to
`react-native-background-geolocation` (TransistorSoft, commercial license); the
queue/API/Redux layers here stay unchanged.

## Push notifications

The spec's notifications are **admin-facing** and are delivered to the admin
dashboard via Socket.IO (and optionally FCM from the backend). The salesperson
app therefore doesn't need to receive pushes. If you later build an admin mobile
app, add `@react-native-firebase/messaging`, register the device token via
`POST /api/me/fcm-token`, and the backend will push to it.
